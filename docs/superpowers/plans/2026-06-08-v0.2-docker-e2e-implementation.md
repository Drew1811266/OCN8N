# v0.2 Docker n8n E2E Validation Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add an opt-in v0.2 validation layer that runs the existing managed n8n workflow lifecycle against a real local Docker n8n instance while keeping ordinary unit tests Docker-free.

**Architecture:** Keep unit tests and real n8n E2E tests in separate Vitest configurations. Add small client and config capabilities needed by the E2E harness, then run deterministic workflow lifecycle tests through the existing core tool functions and one plugin smoke path. The Docker runner owns container lifecycle, readiness checks, secret redaction, and developer-facing diagnostics.

**Tech Stack:** TypeScript, Vitest, tsup, Docker Compose, n8n public API, n8n MCP JSON-RPC endpoint, OpenCode plugin API.

---

## Scope

This plan implements the approved v0.2 spec in `docs/superpowers/specs/2026-06-08-v0.2-docker-e2e-design.md`.

In scope:

- `npm test` stays fast and must not start Docker.
- `npm run test:e2e` starts a disposable local Docker n8n stack and runs real E2E tests.
- E2E verifies create, inspect, preview update, apply update, list, cleanup, API diagnostics, and MCP tool compatibility.
- Tests use deterministic low-risk nodes: Manual Trigger, Webhook, Schedule Trigger, Set, IF, and HTTP Request.
- Plugin smoke verifies OpenCode tool registration and at least one real execute path.

Out of scope:

- Importing or modifying arbitrary existing workflows.
- Updating active workflows.
- OAuth and credential-heavy nodes.
- Project or folder placement.
- Publishing an npm package.

## API And MCP Bootstrap Decision

n8n public API authentication uses `X-N8N-API-KEY`. In normal n8n usage, the API key is created from the n8n UI. The implementation must not hardcode an undocumented API-key seed environment variable. The E2E runner should start n8n, then require `N8N_E2E_API_KEY` unless the implementer verifies a documented local bootstrap mechanism before coding the runner.

The initial runner behavior is:

```text
N8N_E2E_API_KEY is required for real API E2E.
Open http://localhost:5678, create a test API key in n8n, then rerun:
N8N_E2E_API_KEY=<test-key> npm run test:e2e
```

MCP auth is optional in plugin config because local MCP deployments may be open, Bearer-token protected, or unavailable depending on n8n version and configuration. If `N8N_E2E_MCP_TOKEN` is set, the E2E MCP client sends `Authorization: Bearer <token>`. If MCP is unavailable, the E2E suite must fail with a clear capability message instead of silently skipping the check.

function isN8nWorkflowSummary(value: unknown): value is N8nWorkflow & { id: string } {
  if (!value || typeof value !== "object" || Array.isArray(value)) return false

  const workflow = value as Record<string, unknown>
  return (
    typeof workflow.id === "string" &&
    typeof workflow.name === "string" &&
    typeof workflow.active === "boolean" &&
    Array.isArray(workflow.nodes) &&
    typeof workflow.connections === "object" &&
    workflow.connections !== null &&
    typeof workflow.settings === "object" &&
    workflow.settings !== null
  )
}

function isWorkflowListPage(value: unknown): value is { data: Array<N8nWorkflow & { id: string }>; nextCursor?: string } {
  if (!value || typeof value !== "object" || Array.isArray(value)) return false

  const page = value as Record<string, unknown>
  return (
    Array.isArray(page.data) &&
    page.data.every(isN8nWorkflowSummary) &&
    (page.nextCursor === undefined || typeof page.nextCursor === "string")
  )
}

function throwWorkflowListParseError(path: string): never {
  throw new N8nBuilderError("n8n API returned an invalid workflows response.", "N8N_API_PARSE_ERROR", {
    path,
    expected: workflowListExpected,
  })
}
```

```ts
async listWorkflows(): Promise<Array<N8nWorkflow & { id: string }>> {
  const workflows: Array<N8nWorkflow & { id: string }> = []
  let cursor: string | undefined

  do {
    const path = cursor ? `/workflows?${new URLSearchParams({ cursor }).toString()}` : "/workflows"
    let response: unknown

    try {
      response = await this.request<unknown>(path, { method: "GET" })
    } catch (error) {
      if (error instanceof N8nBuilderError && error.code === "N8N_API_PARSE_ERROR") {
        throwWorkflowListParseError(path)
      }

      throw error
    }

    if (Array.isArray(response)) {
      if (!response.every(isN8nWorkflowSummary)) {
        throwWorkflowListParseError(path)
      }

      return [...workflows, ...response]
    }

    if (!isWorkflowListPage(response)) {
      throwWorkflowListParseError(path)
    }

    workflows.push(...response.data)
    cursor = response.nextCursor
  } while (cursor)

  return workflows
}

async deleteWorkflow(workflowId: string): Promise<void> {
  await this.requestNoContent(`/workflows/${encodeURIComponent(workflowId)}`, {
    method: "DELETE",
  })
}

private async requestNoContent(path: string, init: RequestInit): Promise<void> {
  const response = await this.fetchImpl(`${this.baseUrl}${path}`, {
    ...init,
    headers: {
      accept: "application/json",
      "content-type": "application/json",
      "X-N8N-API-KEY": this.apiKey,
    },
  })

  if (!response.ok) {
    throw new N8nBuilderError(
      `n8n API request failed with status ${response.status} for ${path}.`,
      "N8N_API_ERROR",
      {
        status: response.status,
        path,
      },
    )
  }
}
```

## 真实 n8n 实例验证（v0.2）

v0.2 增加了一个显式触发的 E2E 验证层，用本地 Docker n8n 实例验证插件的核心生命周期。默认的 `npm test` 不会启动 Docker，也不会依赖真实 n8n。

### 前置条件

- 已安装 Docker 或 Docker Desktop。
- 能运行 `docker compose`。
- 已在本地测试 n8n 中创建测试用 API Key。

### 运行方式

```bash
N8N_E2E_API_KEY=<你的测试 API Key> npm run test:e2e
```

该命令会启动 `docker-compose.e2e.yml` 中定义的 n8n，等待服务可访问，然后运行 `tests/e2e/**/*.e2e.test.ts`。

### 首次创建 API Key

如果没有传入 `N8N_E2E_API_KEY`，运行器会启动本地 n8n 并提示你打开：

```text
http://localhost:5678
```

在 n8n UI 中完成初始化并创建测试 API Key 后，重新运行上面的命令。

### MCP 配置

默认 MCP URL 为：

```text
http://127.0.0.1:5678/mcp
```

如果你的 n8n MCP 端点需要 Bearer Token：

```bash
N8N_E2E_API_KEY=<你的测试 API Key> N8N_E2E_MCP_TOKEN=<你的 MCP Token> npm run test:e2e
```

### 调试模式

```bash
N8N_E2E_KEEP_ALIVE=1 N8N_E2E_API_KEY=<你的测试 API Key> npm run test:e2e
```

调试模式下，测试失败后 Docker n8n 会保持运行，方便进入 UI 查看 workflow 和日志。普通模式会在结束后执行 `docker compose down --volumes --remove-orphans`，清理测试数据。
```
